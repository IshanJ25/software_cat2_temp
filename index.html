System Modeling
Requirements Specification, DFD, State Machines, Context Models
1 What is System Modeling?

System modeling helps analysts understand functionality and communicate with customers. It creates abstract descriptions of a system from different perspectives.

External Perspective

Shows system context/environment – what lies outside the system

Behavioural Perspective

Shows what the system does – how it behaves

Structural Perspective

Shows system architecture – components and data structures

Data Processing Model

Shows how data is processed at different stages

2 Context Models

Show the operational context of a system — what lies outside the system boundaries. Architectural models show the system and its relationship with other systems.

🧠 Remember: Context model = big picture. "What's around my system?"
3 Data Flow Diagrams (DFD)

A DFD is a visual representation of information flows within a system. It shows how data enters, gets processed, and leaves the system.

SymbolMeaning
Circle / BubbleA Process – transforms data input to output
Curved Line / ArrowData Flow – movement of data
Parallel LinesData Store – where data is stored (e.g., database)
RectangleExternal Entity (Source or Sink)

Levels of DFD

Level 0 (Context)
→
Level 1 (Main processes)
→
Level 2 (Sub-processes)
→
Level 3 (Detail)

Level 0 = entire system as ONE bubble. Each level decomposes further. Inputs/outputs must be preserved across levels (called "leveling" by DeMarco).

🧠 Key Rule: DFD ≠ Flowchart. No decision diamonds! No ordering of events. Just data flow.
4 State Machine Models

Models system behaviour in response to external and internal events. Used for real-time systems. States = nodes, Events = arcs.

Example: Microwave oven – States: Idle → Heating → Done

Example: Traffic Light – Green → Yellow → Red → Green

🧠 Remember: State machines answer "What happens when an event occurs?" – system moves from one state to another.
5 Requirements Elicitation & Validation

Elicitation Techniques

  • Interviews – Direct questions to stakeholders
  • Questionnaires/Surveys – Structured data collection
  • Observation – Watch how users actually work
  • Prototyping – Build a mock to gather feedback
  • Workshops/Brainstorming – Group discussion
  • Document Analysis – Study existing manuals/specs

Requirements Validation

  • Reviews/Inspections – Team reads and checks requirements
  • Prototyping – Validate by showing stakeholders
  • Test-case generation – If you can test it, the requirement is clear

Agile Requirements Management

  • Requirements in User Stories (As a user, I want... so that...)
  • Stored in Product Backlog, prioritized by customer
  • Requirements can change between sprints — Agile embraces change
  • No big requirements document upfront — iterative
Design Concepts & Principles
Abstraction, Modularity, Refinement, Top-Down, Bottom-Up, OO Design
1 Abstraction

A tool that lets a designer consider a component without knowing its internal details.

Functional Abstraction

Module specified by what it does, not how. Basis for Function-Oriented Design.

Data Abstraction

Details of data are hidden from users. Basis for Object-Oriented Design.

🧠 Analogy: You drive a car without knowing how the engine works. That's abstraction.
2 Modularity

Dividing software into separate, independent modules that are integrated to form the complete system.

Properties of a Good Module

  • Well-defined, single specified objective
  • Can be compiled and tested independently
  • Easier to use than to build
  • Simpler from outside than inside

Benefits

  • Easy to understand, test, maintain, and modify
  • Multiple developers can work in parallel
  • Enables code reuse
⚠️ Exam Note: More modules = more communication between them = more complexity. There's an optimal number of modules. "If we modularize more, the effort associated with interfaces rises."
3 Refinement

Stepwise refinement = progressively elaborating details of a design. Start at a high level (abstract) and work down to implementation details.

🧠 Think of it like: First you draw the outline of a painting, then fill in details layer by layer.
4 Design Strategies
Top-Down Approach

Start with main components, decompose into sub-components. Good for new systems.

Main Module → Sub-modules → Details

Bottom-Up Approach

Start with lower-level details, build up to the main system. Good for existing systems.

Details → Sub-modules → Main Module

5 Object-Oriented Design (OOD)

Decomposes a system into interacting objects with well-defined interfaces. Key concepts:

  • Class – Blueprint for objects (attributes + methods)
  • Object – Instance of a class
  • Encapsulation – Data hiding inside objects
  • Inheritance – Child class inherits from parent
  • Polymorphism – Same interface, different behaviour
6 Refactoring

Improving the internal structure of code without changing its external behaviour. Goal: reduce complexity, improve readability, reduce duplication.

🧠 Simple rule: Same output, cleaner code. Like cleaning your room — nothing new added, just better organized.
Coupling & Cohesion
The most important design quality metrics — know ALL types!
★ The Golden Rule

AIM FOR: Low Coupling + High Cohesion

Low coupling = modules are independent. High cohesion = module does one focused thing.

1 Coupling – Worst to Best
Content (WORST) → Common → External → Control → Stamp → Data (BEST) → No Coupling (ideal)
TypeWhat happensAnalogy
Content (WORST) Module directly modifies another's internal code/data Reading someone's diary and changing entries
Common Both modules share a global variable Everyone writes on same whiteboard — chaos
External Modules share externally imposed formats (device, protocol) Two programs must use the same file format
Control One module passes a flag/switch that controls another's logic Telling a cook: "if flag=1 add cheese, if flag=2 add sauce"
Stamp Passes whole object/structure when only part is needed Giving your whole wallet to someone who needs one coin
Data (BEST) Passes only necessary simple parameters Telling taxi: "Take me to airport" — clean, minimal
🧠 Memory trick: C C E C S D → "Can Cats Ever Chase Small Dogs?" (worst → best)
2 Cohesion – Worst to Best
Coincidental (WORST) → Logical → Temporal → Procedural → Communicational → Sequential → Functional (BEST)
TypeWhat it meansExample
Coincidental (WORST) Random unrelated tasks in one module A drawer with socks, batteries, receipts, screwdriver
Logical Similar category tasks, one selected by a flag A module for all "input operations" — but flag picks which
Temporal Tasks grouped because they run at the same time System initialization: clear variables, open files, set counters
Procedural Tasks follow a sequence (not all related but ordered) Recipe steps — must be in order
Communicational All functions operate on same data Everything in a student record about one student
Sequential Output of one part = input to the next Assembly line: cut → polish → paint
Functional (BEST) ONE module = ONE clear responsibility A toaster — it only toasts. That's it.
🧠 Memory trick: C L T P C S F → "Cats Love To Play, Cats Stay Friendly" (worst → best)
3 Quick Identification Tricks
  • Sees global variable shared? → Common Coupling
  • Module changes another module's code/data? → Content Coupling
  • Passes flag/boolean to control logic? → Control Coupling
  • Passes whole struct/object but uses part? → Stamp Coupling
  • Passes only needed parameters? → Data Coupling
  • Module depends on interface? → Low coupling (good!)
Architectural Design
System structure, architectural styles, and patterns
1 What is Architectural Design?

The process of identifying sub-systems and their interactions. It's an early design stage — the link between specification and detailed design.

Key Architectural Attributes (Performance vs Quality)

QualityArchitectural Solution
PerformanceMinimize sub-system communication; localize operations
SecurityLayered architecture; critical assets in inner layers
SafetyIsolate safety-critical components
AvailabilityInclude redundant components
MaintainabilityFine-grained, self-contained components
2 Architectural Styles / Patterns
Repository Model ▼

All sub-systems share data through a central repository/database.

✅ Advantages: Efficient data sharing, centralized management (backups, security), published schema.

❌ Disadvantages: All must agree on data model, difficult to distribute, expensive evolution.

Example: IDE tools (compiler, editor, debugger) all sharing a code repository.

Client-Server Architecture ▼

Distributed model: clients call services provided by servers over a network.

✅ Advantages: Easy to add/upgrade servers, effective use of network, scalable.

❌ Disadvantages: No shared data model, redundant management per server, no central service registry.

Example: Film/photo library server serving multiple clients.

Layered (Abstract Machine) Architecture ▼

System organized as layers, each providing services to the layer above.

Presentation Layer
↔
Business Logic
↔
Data Layer

✅ Advantages: Incremental development; changing one layer only affects adjacent layer. Good for security (critical assets in inner layers).

❌ Disadvantages: Difficult to cleanly separate layers in practice.

MVC (Model-View-Controller) ▼
Model

Data and business logic. The core data.

View

What the user sees. UI presentation.

Controller

Handles user input and updates Model/View.

Best for: Web applications (used in Rails, Django, Spring MVC)

Supports multiple presentations of the same data.

Microservices Architecture ▼

System built as small, independent services each running its own process and communicating via APIs.

✅ Advantages: Independent deployment, scalability per service, fault isolation, technology flexibility.

❌ Disadvantages: Complex communication, distributed system challenges, operational overhead.

vs Monolithic: Monolith = everything in one big app. Microservices = each feature is its own app.

3 Control Models
Centralised Control

One sub-system controls all others. Types: Call-Return (top-down hierarchy) and Manager (concurrent systems).

Event-Based Control

Sub-systems respond to events. Types: Broadcast (any handler can respond) and Interrupt-driven (real-time, fast response).

4 Modular Decomposition Models
Object Model

Decompose into loosely coupled objects with defined interfaces.

Data-Flow (Pipe & Filter)

Functional modules that transform inputs to outputs. Like UNIX pipes.

Detailed Design
Transform Mapping, Transaction Mapping, DFD to Architecture
1 Structured Design Overview

A data flow-oriented design method that maps DFDs into software architecture. The 6-step process:

1Establish the type of information flow (transform or transaction)
2Indicate flow boundaries
3Map DFD into program structure
4Define control hierarchy
5Refine structure using design heuristics
6Refine and elaborate architectural description
2 Transform Flow vs Transaction Flow
Transform Flow

Data flows linearly through the system — input → processing → output. Most common.

Like: Read file → Process data → Write output

Transaction Flow

A single data item can trigger multiple possible paths depending on the transaction type.

Like: A bank transaction — could go to Deposit, Withdraw, or Transfer path

3 Transform Mapping Steps
  • Review fundamental system model – context diagram (Level 0)
  • Review & refine DFDs – produce more detail (Level 2, 3)
  • Determine flow type – transform or transaction
  • Isolate the transform center – find incoming/outgoing flow boundaries
  • First-level factoring – map to top-down control structure with 3 controllers:
    • Incoming Information Processing Controller
    • Transform Flow Controller
    • Outgoing Information Processing Controller
  • Second-level factoring – map individual DFD bubbles to modules
  • Refine – apply functional independence, cohesion, minimal coupling
🧠 Remember: Transform center = the "heart" of processing. Incoming = reading/converting input. Outgoing = formatting output.

3 Nielsen's 10 Usability Heuristics
  • Visibility of system status – Always show what's happening (progress bars, loading)
  • Match between system and real world – Use user's language, not tech jargon
  • User control and freedom – Undo, redo, easy exit
  • Consistency and standards – Same words/actions mean the same thing
  • Error prevention – Design to prevent errors before they happen
  • Recognition over recall – Show options rather than making users remember
  • Flexibility and efficiency – Shortcuts for experts, still usable for novices
  • Aesthetic and minimalist design – No irrelevant information
  • Help users recognize, diagnose, recover from errors – Clear error messages
  • Help and documentation – Easy to search, focused on user's task
4 Information Presentation
Digital Display

Compact, precise values. Use when exact numbers matter.

Analogue Display

Easier to get "at a glance" impression. Shows relative values and trends.

Colour Use Guidelines

  • Don't use too many colours (<5 ideally)
  • Use colour consistently (same colour = same meaning)
  • Design for monochrome first, then add colour
  • Avoid colours that clash; be aware colour-blind users exist
  • Use colour change to show status change
5 Accessibility Features
  • Screen reader support – Alt text for all images
  • High contrast mode – Clear text visibility
  • Keyboard navigation – No mouse required
  • Text size control – Allow font scaling
  • Voice input/output – For motor/visual impairment
  • Descriptive error messages – Tell users HOW to fix errors
6 Error Messages Design Rules

Good error messages must be: Polite, Concise, Consistent, Constructive

❌ BAD

"FATAL ERROR: Illegal operation at 0x004a3f22"

✅ GOOD

"Your password must be at least 8 characters. Please try again."

Software Testing
Fundamentals, Types, Test Design, Reviews & Inspection
1 Testing Fundamentals
Validation Testing

Does the software do what the customer wants? Shows software meets requirements.

Defect Testing

Finding bugs. A successful test is one that exposes a defect.

⚠️ Key Statement to write: "Testing can show the presence of faults. It cannot prove there are no remaining faults. Exhaustive testing is impossible."
2 Testing Types (Hierarchy)
Unit/Component Testing Individual functions/methods/classes tested in isolation. Done by developer.
↓
Integration Testing Testing components combined together. Tests interfaces between components.
↓
System Testing Complete system tested. Done by independent team. Based on specification.
↓
Release/Acceptance Testing Testing release for customer. Black-box. Confirms customer requirements met.
🧠 Who tests what: Developer → Component; Independent team → System & Release.
3 Integration Testing Approaches
Top-Down Integration

Start with skeleton/main framework, add components. Better for architectural validation and early demo.

Bottom-Up Integration

Start with infrastructure components, add functional ones. Easier test implementation.

🧠 Rule: Always integrate incrementally to simplify error localization.
4 Test Case Design Approaches
Equivalence Partitioning (Partition Testing) ▼

Group inputs into partitions/classes where all members behave the same way. Test one value from each partition.

Example: Password field (6-10 chars):

  • Valid partition: 6 to 10 characters → test with 8 chars
  • Invalid partition (low): 0 to 5 → test with 3 chars
  • Invalid partition (high): 11+ → test with 15 chars
  • Boundary values: 5, 6, 10, 11 (test the edges!)
Structural Testing (White-Box) ▼

Uses knowledge of program code to derive test cases. Goal: exercise all program statements.

Path Testing: Ensure every path through the code is executed at least once. Uses a flow graph (nodes = decisions, arcs = control flow).

Cyclomatic Complexity = E − N + 2P

E = edges, N = nodes, P = nodes with exit points. Gives minimum number of test cases needed.

Black-Box Testing (Functional) ▼

Tests based on specification only — no knowledge of internal code. Used in Release Testing.

Testers just know: given input X, expected output is Y.

5 Special Testing Types
TypeWhat it tests
Performance TestingSystem performance under increasing load — find the breaking point
Stress TestingPush system beyond maximum design load — test failure behaviour
Interface TestingFind defects in interfaces between components — check parameter passing
Object Class TestingTest all operations, attributes, states of an object/class
6 Reviews, Inspection & Auditing
Review

Team examines software artifacts (requirements, design, code) to find defects. No testing involved.

Inspection

Formal review with defined roles (moderator, reviewer, author). Follows a structured process with defect checklist.

Audit

Independent check to ensure processes and standards are being followed.

Test Automation

Use tools (e.g., JUnit) to automatically run tests. Reduces cost. Most testing workbenches are open systems.

🧠 Key benefit of reviews: Find defects early (before coding). Cheaper to fix at requirements stage than at testing stage.
🎯 Exam Q&A
Ready-to-write answers for likely exam questions — based on your exercise sheets
Q1 Which architecture for a University LMS with thousands of users?

Answer: Microservices Architecture

  • Scalability: Each service (videos, assignments, exams) scales independently based on demand
  • Fault isolation: If the video service fails, exam system still works
  • Independent deployment: Teams can update the assignment service without affecting others
  • Technology flexibility: Video streaming can use different tech from the exam module
Tip: Also acceptable: Client-Server for simplicity, or MVC for structure. But microservices is best for "thousands of simultaneous users."
Q2 Why is consistency important in UI design?
  • Reduces learning time – Users learn once and apply knowledge everywhere
  • Prevents errors – Different patterns for same action confuse users
  • Builds user confidence – Predictable behaviour makes users feel in control
  • Supports recognition over recall – Users recognize familiar patterns
Q3 Classify coupling: worst to best from given examples

Given (from your exercise PDF):

  1. Module modifies another's private data → Content Coupling (WORST)
  2. Two modules share a global variable → Common Coupling
  3. Module passes a control flag → Control Coupling
  4. Module passes only necessary parameters → Data Coupling
  5. Module depends on an interface → No/Low Coupling (BEST)

Ranking (worst→best): 1 → 2 → 3 → 4 → 5

Q4 Identify usability heuristics violated: "Important messages in small gray text at bottom"
  • Visibility of system status – Important info must be visible, not hidden
  • Aesthetic and minimalist design – Important content should NOT be buried with irrelevant content
  • User guidance – Users need clear guidance, small gray text is unreadable
  • Accessibility – Violates accessibility for visually impaired users
Q5 Layered architecture: Identify style and benefits

The style is: Layered Architecture (Abstract Machine Model)

  • Maintainability: Changing one layer only affects the adjacent layer, not the whole system
  • Reusability: Each layer can be reused with different upper/lower layers
  • Separation of concerns: Presentation logic separate from business logic and data
  • Testability: Each layer can be tested independently

Q7 3 UI techniques to reduce wrong travel date entry
  • Date picker / Calendar widget – Visual selection prevents format errors
  • Validation with real-time feedback – Highlight invalid dates immediately (past dates, departure after return)
  • Confirmation screen – Show full trip summary before final booking so users can verify
Q8 Monolithic → Microservices: What's the problem and solution?

Problems with Monolith at scale:

  • Single codebase becomes too large and hard to maintain
  • Any change requires redeploying the entire system
  • One component failing can crash the whole application
  • Cannot scale individual parts — must scale everything

Microservices solution:

  • Each feature (cart, payments, user auth) becomes its own service
  • Independent deployment and scaling per service
  • Teams can work independently on different services
  • Fault in one service doesn't bring down others

⚡ Last-Minute Key Facts to Remember
  • Low coupling + High cohesion = GOOD design
  • Content coupling = WORST; Data coupling = BEST
  • Coincidental cohesion = WORST; Functional cohesion = BEST
  • DFD ≠ Flowchart (no decisions, no ordering in DFD)
  • Testing proves presence of bugs, NOT absence
  • Exhaustive testing is impossible
  • Component testing = developer; System testing = independent team
  • Black-box = no code knowledge; White-box = uses code structure
  • Equivalence partitioning: test one value per partition + boundary values
  • Microservices = independent services + API communication
  • Layered architecture = change one layer, only adjacent layer affected
  • MVC = Model (data), View (UI), Controller (logic/input)
  • State machine = states as nodes, events as arcs
  • Refactoring = better code structure, same external behaviour
